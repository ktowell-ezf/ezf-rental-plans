Rental Plans — Admin Configuration

📄 PRD: Rental Packages — January 2026 Draft ✏️ Prepared by K. Towell 🗓 May 2026
Wireframe scope: This wireframe covers the Admin Configuration experience — the standalone Rental Plans admin page where facility admins create, view, edit, and archive Rental Plans. Sales, schedule deduction, client profile, reporting, and Self Service flows are out of scope for this wireframe (see Section 5).

Rental Plans List

Story 1 ✓ Story 10 partial
Page Tabs ▶
  • Active — shows all non-archived plans with count badge
  • Archived — shows archived plans with count badge; switching tab resets filters and selection
Toolbar ▶
  • Search field — matches against any visible plan detail (name, purchase model, pricing model, description, applied rate names)
  • Filters button opens popover: Status · Purchase Model · Pricing Rules multiselect · Standard Price range · Member Price range · Clear filters
  • Results count ("X plans") — replaced by "X selected" when rows are checked
Bulk Actions ▶
  • Always visible in toolbar; disabled until rows are selected
  • Active tab: single Archive button — because there is only one available bulk action, no dropdown is needed; enabled on selection and pulses briefly to draw attention on first selection
  • Archived tab: Actions ▾ dropdown containing Restore and Delete — a dropdown is used here because there are two available bulk actions; this keeps the toolbar clean while making all options accessible
  • Deselect All button appears when rows are checked
UX pattern: When there is only one bulk action available, it is shown as a direct button. When there are two or more, they are consolidated into a dropdown. This avoids cluttering the toolbar while keeping all actions reachable. OPTIONAL CHANGE Affects global grid component
Grid Columns & Actions ▶
  • Checkbox · Name · Status badge · Purchase Model · Standard Price · Member Price · Expiration · Pricing Rules
  • Pricing Rules cell shows applied rate names when rates are assigned; shows Custom Rule (italicized) when the plan uses ad hoc pricing with no applied rates
  • Row actions (⋯): View, Edit, Archive — or Restore / Delete on Archived tab
  • Add Rental Plan button in page header opens the Add Rental Plan side panel
Satisfies Story 1 list view requirements: columns, filter/search, archive. Story 10 entry points (View + Edit row actions) are present.

Naming note: Plan status values use Available and Unavailable rather than Active / Inactive to avoid confusion with the Active / Archived tab names. "Active" at the tab level describes whether a plan has been archived; "Available" at the status level describes whether the plan is currently offered for sale. Using distinct terminology prevents ambiguity for both staff and future development.

How custom rates are tracked: When a plan is saved with custom pricing unique to that rental plan, the rate is created and displayed as a read-only entry in the Rental Rates grid — it does not appear as a selectable option when searching applied rates on the Add or Edit Rental Plan panels. In the Rental Rates grid, the Rental Type value for these rows will be Rental Plan: [plan name]. Rather than Edit/Delete actions, these rows will show a button that navigates directly to the associated rental plan. A design update to the rental rates grid can be provided if approved.

Open Questions — Design Impact

Q2 · Addressed in wireframe
What level of audit logging is required?
History tab added to Edit panel with chronological change log. Scope and retention policy TBD with Engineering.
Q1 · Needs answer
What permission levels are needed for selling vs. configuring?
No permission-gating shown in this wireframe. Will need to be layered in once permission model is defined.
Q3 · Needs business decision
Can an active (sold) plan configuration be edited?
Wireframe allows editing all fields. The recommended approach is that edits to a sold plan take effect only at the next renewal — not retroactively. This means the client must be notified of the upcoming change and given the opportunity to cancel renewal before it applies. This should be confirmed as a business rule before engineering estimates Story 10.
Q4 · Needs answer
Balance restoration on delete — automatic or prompted?
Not addressed in this wireframe (Schedule flow). Behavior decision needed before Story 7 design begins.
Q5 · Needs answer
Reporting — new report or update existing?
Not addressed in this wireframe. Options A, B, C from PRD remain open.
Q6 · Resolved
Terminology differences from Package Plans?
"Rental Packages" has been decided across departments to be referred to as Rental Plans. This is reflected throughout the wireframe. Remaining term decisions (Check-In vs. Attended, etc.) apply to Schedule and Client Profile flows and remain open for those workstreams.
7

Design Decisions for Product Owner Review

Decision 1 · Story 10
Revert Changes vs. Save Confirmation
The PRD specifies a confirmation prompt before saving changes. The wireframe implements a Revert Changes button instead — it appears in the footer once any field has been edited, and shows an inline confirmation before reverting to last saved state. This avoids interrupting the save flow. Confirm preferred pattern before handoff.
Decision 2 · Q2
History Tab as Audit Trail
A History tab was added to the Edit panel proactively to address Q2. It shows a read-only chronological log of field changes with user, timestamp, and old → new values. Confirm whether this satisfies the audit trail requirement or whether a more robust solution (exportable log, separate page) is needed.
Decision 3 · Story 1
Applied Rates vs. Custom Pricing
The Pricing tab offers both Applied Rates (sourced from the Rental Rates admin page) and Custom Pricing modes via a segmented control. The PRD does not explicitly define this as a UI pattern. Confirm this aligns with product intent.
Decision 4 · Story 1
Rate Rules Display in Plan Summary
Pricing rules attached to each Applied Rate are shown collapsed by default behind an expandable ▸ N rules toggle in the Plan Summary view, consistent with the Edit/Create panels. Confirm this is the right level of detail for the read-only summary.
